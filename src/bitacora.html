📋 Decisiones de Diseño

Estructura y Organizacion

Decidimos colocar todo dentro de una carpeta /src para una mejor organización. Dentro de ella, tenemos el home (index.html), la bitacora, y carpetas como /js, /css, /img, /integrantes y /assets. Creemos que estos nombres son autoexplicativos, algo que en programación nos parece util: la capacidad de una "variable" (en este caso nombres de archivos y carpetas, expliquen por si solos su funcionalidad o utilidad).

Con respecto a la "sección" de cada uno, lo hicimos simple: una carpeta /integrantes con una carpeta para cada persona. Podríamos haberlo hecho sin una carpeta por integrante, pero lo decidimos así para darnos libertad a cada uno de agregar y modificar lo que quería dentro de su carpeta.

Estilos

Decidimos usar una base para el home, la bitacora y todas las paginas de los integrantes (En esta ultima lo consideramos estilo fallback). La idea es tener esta base solo en el home y la bitacora, y que cada uno cree su estilo y secciones, para practicar y hacer libremente, ya que consideramos que es la mejor forma de aprender.

🛠️ Dificultades Encontradas y Soluciones

Responsive Design

Desafío: Lograr que el diseño se adapte correctamente a los tres breakpoints requeridos (400px, 900px, 1200px) sin que se rompiera la estructura.

Solución: Implementamos un sistema flexible con CSS y utilizmos media queries específicas para cada breakpoint. Probamos el diseño en múltiples dispositivos (simulados en el navegador) para asegurar la consistencia.

La IA es muy potente

Desafío: Usar el codigo del único prompt que realizamos a la IA de forma consciente: todo el código que presentamos es codigo que leimos y comprendimos.

Solución: En primer lugar fue leer el output que nos generó la IA, una vez que lo visualizamos decidimos ponernos a comprender lo que hizo tanto con estilos, como con archivos de javascript y de html. Para ser sinceros con la entrega, quitamos funciones que eran muy complejas y dificiles de leer. Si bien funcionaba todo a la perfección, decidimos sacar las cosas que eran dificiles de entender para ser honestos intelectualmente: el codigo entregado es codigo que entendemos. Como es un trabajo practico con el objetivo de aprender, consideramos entregar algo quizás mas sencillo, pero que fué leido y comprendemos

📝 Reflexiones del Equipo

Con el pasar de los días podemos llegar a muchas conclusiones, algunas pueden ser:

La IA es una herramienta genial optimiza nuestro tiempo y nos da algo que es funcional en cuestión de segundos. Así y todo consideramos que es nuestro deber comprender cada linea que usemos del código, en primer lugar para ser mejores desarrolladores y en segundo lugar para refactorizarla si es necesario, para crear codigo modular y mantenible. Si solo copiamos y pegamos, vamos a generar código inmantenible, y probablemente con bugs

Usar Github o algun tipo de repositorio remoto en vez de Google Drive o enviarnos los archivos es una obligación y NO es negociable, siempre debemos utilizarlo ya que al trabajar en grupo, puede que varios estemos trabajando en paralelo, sin algo como github nos costaría muchisimo tener el codigo actualizado constantemente para seguir haciendo cambios, además de poder tener un seguimiento de que cambios se hicieron en cada momento.
